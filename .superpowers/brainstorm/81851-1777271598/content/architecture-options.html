Sola Editor: Typora Parity Re-architecture

To achieve complete Typora parity, we need to unify the document model and the GPUI rendering layer. How should we approach the fundamental Text Layout architecture?

A

The Zed Way (Custom Element + TextSystem)

Recommended. Render everything inside a single gpui::Element. Manually map AST tokens to gpui::TextRun and use the low-level TextSystem to shape and layout text. This is what Zed does for its code editor. It provides pixel-perfect hit-testing and caret positioning, but requires manually handling line wrapping, inline images (math), and cursor navigation math.

B

The GPUI Flex Way (Div Soup)

Continue with the current approach: map every AST node to a tree of GPUI div() elements. Rely on GPUI's Flexbox engine for layout. Pros: Easy to insert inline math/SVG. Cons: Hit-testing (clicking exactly between characters in a wrapped paragraph) and caret rendering are notoriously difficult and buggy because GPUI's DOM isn't a text engine.

C

Hybrid: Blocks as Flex, Paragraphs as TextSystem

Use GPUI flexbox to stack blocks (Paragraph, CodeFence, Quote) vertically. But for the contents of a Paragraph, use the low-level TextSystem. This simplifies vertical navigation but still requires custom hit-testing logic inside the paragraph for text vs inline math.
